# 개발 백로그 (SpecFlow AI / Deal-Closer)

> 이 문서는 `docs/SPEC.md`(기획 기준)와 현재 구현 상태를 대조해 도출한 **개발 백로그**입니다.
> 핵심 분리 축: **① MVP 데모 필수(실제 동작 필요)** vs **② 목업으로 시연 가능(추후 실개발)**.
> 규모: S(작음) / M(보통) / L(큼). ID(A1·B2…)는 트래킹용 식별자입니다.

## 현재 구현 현황 (백로그의 출발점)

| 영역 | 상태 | 근거 |
|---|---|---|
| 24개 화면 페이지 | ✅ 전부 존재 | `src/app/**/page.tsx` (14) + 상태화면 |
| 문서 CRUD | ✅ 실제 DB | `/api/documents` GET/POST/PATCH/DELETE |
| 크레딧 차감 | ✅ 실제 DB | `/api/generate` 트랜잭션 (10크레딧/건) |
| **AI 생성** | ⚠️ 목업 | 정규식 `inferType()` + 빈 예시 라인 1줄, 실제 LLM 없음 |
| **이메일 발송** | ⚠️ 목업 | `EmailLog` 기록 + `status=SENT`만, 실제 전송·PDF 없음 |
| **메일 OAuth** | ⚠️ UI만 | `connect-button` UI, 실제 OAuth 플로우 없음 |
| **인증** | ⚠️ 데모 고정 | `session.ts` 홍길동/admin 하드코딩 |
| 라인아이템 편집 저장 | ⚠️ 공백 | 문서 PATCH가 `items`(라인아이템) 미반영 |
| 메일 템플릿 | ✅ 실제 DB (MVP 추가) | `EmailTemplate` CRUD(`/api/email-templates`) + 발송폼 불러오기·치환 변수 |
| 메일 서명 | ✅ 실제 DB (MVP 추가) | `User.signature` + `/settings/email` 편집, 발송폼 본문 하단 추가 |

## ① MVP 데모 필수 — 실제로 동작해야 함

> 핵심 해피패스(**자연어 생성 → 편집 → 발송**)가 시연 중 티나게 깨지지 않으려면 진짜로 작동해야 하는 항목들입니다.

| # | 백로그 | 규모 | 왜 필수인가 |
|---|---|---|---|
| A2 | **라인아이템 편집 저장** — 에디터 수량·단가 수정 → `DocumentItem` upsert + 금액 서버 재계산(VAL_*) | M | 에디터에서 고친 값이 저장 안 되면 데모 중간에 바로 드러남. 현재 문서 PATCH는 헤더만 저장 |
| C4 | **수신자 이메일 검증** — 발송 폼 이메일 형식·다중 수신자 파싱(VAL_*) | S | 발송 단계 최소 신뢰성. 잘못된 입력에 폼이 무반응이면 데모가 불안정해 보임 |
| C1 | **문서 상태 전환 완성** — DRAFT→SENT(발송 시 자동)→COMPLETED(계약완료 버튼) | S | 라이브러리 상태 탭이 실제로 움직여야 스토리가 완성됨 |
| F1 | **상태 화면(최소)** — `error.tsx` · `not-found.tsx` · 주요 라우트 `loading.tsx`(STATE_*) | S | 데모 중 예외 시 흰 화면/크래시 방지 |
| F2 | **에디터 이탈 경고** — 미저장 상태 이탈 시 컨펌(STATE_*) | S | 편집→발송 전환 시 데이터 유실 방지, 기획 정책 항목 |
| A1 | **AI 문서 생성 실개발(LLM)** — 프롬프트 → 문서 타입·제목·라인아이템 구조화 생성 | L | 제품의 핵심 가치. **단, 순수 UI 데모라면 "정교한 템플릿 목업"으로 대체 가능**(아래 참고). 실제 AI 시연이 목표면 필수 |

> **A1 판단 가이드**: 투자자/고객에게 "AI가 진짜 생성한다"를 보여야 하면 → ①에 두고 실개발. 화면 흐름만 보여주는 UI 데모면 → 현재 목업을 더 그럴듯한 템플릿으로 다듬어 ②로 내려도 됩니다.

## 권장 착수 순서

1. **①(MVP 데모 필수)를 전부 완료** — 특히 `A2 → C4 → C1 → F1/F2`. 여기까지면 해피패스 데모가 안정적으로 돌아갑니다.
2. **A1(AI 실개발)** — 데모 목표가 "실제 AI"라면 최우선, "UI 흐름"이면 목업 유지.
3. 이후 SaaS 성립 요건: **E1(인증) → D1(결제) → A3·A4(PDF·발송)** 순으로 ②에서 승격.

> 규칙: 기능이 SPEC 정책(`AUTH_*`·`VAL_*`·`FORM_*`·`STATE_*`·`ACC_*`·`legal-*`·`COPY-TONE`)과 얽히면 구현 시 해당 정책을 함께 반영합니다. SPEC 변경이 필요하면 `docs/SPEC.md`·`prisma/schema.prisma`·`AGENTS.md`를 함께 갱신합니다.
